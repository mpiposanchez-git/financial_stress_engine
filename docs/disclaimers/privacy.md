# Privacy Policy

## Summary

This tool is designed to minimise personal data processing.

- **Authentication is provided by Clerk** (third-party identity provider).
- The simulation engine is designed to run **in-memory** and **does not persist** your stress test inputs or outputs in a project database.
- This Proof of Concept does **not** use analytics, tracking pixels, session replay, or advertising identifiers.
- The API keeps only **aggregate operational telemetry counters** (for reliability monitoring), without storing user IDs or request payload values.

**Important:** Like most web services, some **infrastructure-level logs** (e.g., web server access logs) may be generated by hosting providers. See “Infrastructure Logs” below.

---

## Authentication (Clerk)

Some application routes require sign-in. Authentication is managed by Clerk.
Authentication is **always enforced** for protected API routes in this PoC (no runtime auth-disable/bypass mode).

When you sign in:
- Clerk may process account information and authentication/session metadata under its own policies.
- This includes your sign-in email and may include an email alias/anonymous email account generated by Clerk for identity protection flows.
- The API validates JWT bearer tokens issued by Clerk to authorise requests.
- The project does **not** maintain its own user database of profiles (no separate user table in a project database in this PoC).

"The API may use your Clerk user identifier (the JWT 'sub' claim) in-memory for rate limiting and abuse prevention; this identifier is not stored in a project database by this PoC."

---

## What Personal Data This Project Processes

### Data processed by Clerk (third party)
Clerk may process:
- Email address used for login (including Clerk-provided anonymous/relay email aliases where applicable)
- Session/authentication metadata required to operate sign-in securely

This project relies on Clerk for identity and does not replicate user profiles into a separate project database.

### Data processed by this project (minimised)
The project itself is designed **not** to request or store household PII. In particular, the project does not intentionally request or store:
- Name
- Address or postcode
- National Insurance number
- Date of birth
- Bank account credentials or Open Banking tokens
- Uploaded documents (statements, payslips, etc.)

---

## Household Inputs and Results (Stress Test Data)

The numerical values you enter into the stress test form are:
- Used **in-memory** to compute your results.
- **Not persisted** to any project-owned database or file storage.
- **Not intentionally written** to application logs.
- Discarded after your results are returned.

This tool is designed for **manual entry** of numbers only (no document uploads).

---

## No Bank Linking / No Uploads

- This tool does **not** connect to banks, Open Banking, payment providers, or financial institutions.
- You do not upload bank statements, payslips, tax returns, or other documents.
- Inputs are entered manually as numerical values.

---

## Cookies

This tool does not use tracking or analytics cookies.

Clerk may use cookies or browser storage for authentication/session management (necessary for sign-in).

---

## Infrastructure Logs (Hosting Providers)

Even if the application does not store your simulation inputs, hosting providers may generate standard operational logs, such as:
- IP address
- User agent / device/browser information
- Timestamps and requested URL paths
- HTTP status codes

These logs are used for security and operational reliability (e.g., troubleshooting, abuse prevention) and are controlled by the hosting providers according to their policies.

Application-level operational logging is limited to request metadata (e.g., method/path/status/timing). The application is designed not to log request bodies or authentication bearer tokens.

The repository also includes automated backend tests that fail CI if sensitive logging patterns (for example request bodies or bearer token logging) are introduced.

---

## Aggregate Operational Telemetry

To monitor service health, the API maintains in-memory aggregate counters such as:
- deterministic run count
- Monte Carlo run count
- PDF export count
- error count
- coarse runtime buckets

These counters are aggregate only and are designed to exclude scenario payload values and personal identifiers. The telemetry endpoint is restricted to admin access using an environment-configured admin token.

---

## Security Notes (Operational Commitments)

We aim to minimise privacy risk by:
- Not storing stress test inputs/outputs in a project database
- Avoiding analytics/tracking scripts
- Avoiding logging of request payloads that could contain user-entered values
- Avoiding logging of authentication tokens

If a security incident or unintended data collection is suspected, we will investigate and remedy the issue.

---

## UK GDPR Considerations

Under the UK GDPR and the Data Protection Act 2018:
- This project’s own processing is designed to be minimal and non-persistent for stress test inputs.
- Clerk acts as a third-party provider for authentication and processes account-level data under its own legal terms and policies.

If you believe personal data has been inadvertently collected or processed, contact us so we can investigate.

---

## Third-Party Services

This tool integrates with:
- **Clerk** (authentication)

This tool does not integrate with:
- third-party analytics
- advertising networks
- data enrichment services

## Changes to This Policy

This privacy policy may be updated from time to time. Changes will be reflected in the repository and versioned.
